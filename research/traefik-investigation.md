## What broke

1. **Zombie task.** Before restart, the container was `Up 2 weeks (unhealthy)`, but `docker inspect` showed `"task e18bf25...  not found"` for every recent health-check probe. The healthcheck command is `traefik healthcheck --ping`, which does a `docker exec` — and the exec kept failing with "task not found." That's consistent with containerd thinking the container task is gone while the docker daemon still believes the container is "running." Since docker-proxy on :443 only forwards after the container task exists, external TCP to :443 got connection-refused (I confirmed from the client side — it was refused, not timed out).
2. **Restart policy is `unless-stopped`** but docker didn't restart because from docker's view the container was still "running," just unhealthy. Health status doesn't trigger restart by itself.
3. **Why the task disappeared is unknown.** No OOM (memory unlimited, host not under pressure). No SIGKILL in journal that I found. Possible causes to investigate:
   - Known traefik v3.6 / docker / containerd interaction bug — check upstream issues for "task not found" with traefik healthcheck
   - A runc/containerd upgrade on the host (journal shows kernel unchanged for weeks — check apt history for runc/containerd/docker.io timestamps near 2026-04-06 through -08 when the unhealthy state likely set in)
   - The `--api.insecure=true` + file provider combo isn't inherently unstable, but worth diffing against the Ansible source to see if there was a recent change

- Inventory: `/home/mmcdonnell/workspace/McHomeLab-Inventory/hosts.yml` — unifi host, `services.traefik` block including `external_routes` (unifi-os and uisp).

### What the repo adds beyond the original write-up

- **UniFi OS and UISP installers are disabled in Ansible.** `roles/unifi-os/tasks/main.yml` and `roles/uisp/tasks/main.yml` download installers to `/opt/unifi-os/installer` and `/opt/uisp/install.sh` but the execution tasks are commented out (commit `0f0860a`, 2026-03-13, "Comment out installer execution tasks for manual install workflow"). **Pasta port-forwarding (container ns :443 → host :11443) is outside Ansible entirely.** Any future investigation that expects Ansible to describe the full unifi stack will be disappointed — the chain stops at the installer download.
- **Two-provider pattern.** Traefik loads the docker provider (auto-discovery of containers on the same compose network) AND the file provider (the `traefik-external.yml` with route definitions). On unifi the docker provider is effectively empty — there are no other services in the compose project that traefik fronts. Every meaningful route comes from the file provider. This is inventory-driven: `external_routes` in hosts.yml produces the file-provider config.
- **`--api.insecure=true` is hardcoded in the template, published on host port 9880.** There is no auth and no traefik routing in front of it. Worth confirming this port is blocked by the UXG firewall from anything outside the management VLAN.

### Healthcheck tradeoff (answer to original question #3)

**Short answer: switching from `traefik healthcheck --ping` (exec) to an HTTP probe would not have prevented this failure.** Docker's `HEALTHCHECK CMD` always uses `docker exec` to run the test command, regardless of whether the command is `traefik healthcheck` or `wget http://localhost:8080/ping`. When the containerd task disappears, every exec fails with "task not found" — the probe form doesn't matter.

More importantly: **the restart policy (`unless-stopped`) does not react to health status**. An unhealthy container that is still "running" from docker's view is never restarted by docker itself. That's the real gap. The exec vs. HTTP probe question is cosmetic compared to this.

Two real options, roughly in order of preference:

1. **Investigate root cause before adding machinery.** Correlate apt history on unifi for `containerd`, `runc`, or `docker.io` upgrades between 2026-04-06 and 2026-04-08 (the likely window when the task died). Check `journalctl` for SIGKILL or containerd errors from that window. Look for upstream traefik v3.6 / containerd bugs. If a specific bug is identified, pinning to a known-good version is the cleanest fix.
2. **Add an autoheal sidecar** (e.g., `willfarrell/autoheal`) as a service in the same compose project. It watches for `Health: unhealthy` containers and restarts them. This is defense-in-depth regardless of root cause and fits the repo's pattern of adding services through `hosts.yml`. Trivial to template.

**Not recommended:** switching the healthcheck command to HTTP for "robustness." It would change nothing about the zombie scenario, and the repo's convention for infrastructure services (step-ca, apt-cacher-ng) is already mixed — traefik's `--ping` is purpose-built for this exact check and the binary is guaranteed to be in the image.
